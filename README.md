# Description

**Call_me_maybe** is a constraint generation tool for Small Language Models focused on deterministic Function Calling.

**Goal**
To guarantee that a local language model outputs strictly formatted and valid JSON objects that match predefined function signatures based on natural language user requests.

**Overview**
The program intercepts the decoding process of a language model. It takes a user prompt and a JSON array of available functions (including descriptions and expected parameters) as input. By applying algorithmic constraints during the logit decoding phase (such as dynamic token masking and structural state management) the system prevents syntax errors and text hallucinations. The final output is a reliable JSON object containing the prompt, the selected function's name and the extracted arguments required to execute the user's intent.

# Instructions

### Installation

You have to execute the ```make install``` command. It will installs all the required dependancies for this program in a virtual environment.

### Execution

The source code is provided with a Makefile containing the following rules:

```bash
make : Provides all commands available.

make install : Installs all dependancies needed in a virtual environment.

make run : Executes the program.

make clean : Deletes all cache directories and compiled files generated by Python interpreter.

make clean-all : Delete the virtual environment.

make lint : Test mypy and flake8 linters.

make lint-strict : Same as above but in strict mode.
```

### Usage

There are demo JSON files, but if you want to use your own, you will need to use the ```uv run python3 -m src``` command and add some flags:

```--functions_definition <function_definition_file_path>``` for your own functions definitions in a JSON file.

```--input <input_file_path>``` for your own prompts in a JSON file.

```--output <output_file_path>``` for your own output file.

# Thread synchronization mechanisms

## The mutexes :

**Resource locking :** Each dongle has its own mutex (dongles[i].lock) guaranteeing exclusive access to the resource.

**Data protection (Data Races) :** Other mutexes isolate sensitive variables. ```lock_coder_time``` protects the last compilation time, ```lock_coder_request``` protects the access to the request_time value, ```lock_coder_data``` secures the access to standard output and ```lock_coder_finish``` secures the access to the finish variable.

**Global state :** ```lock_sim_state``` secures the reading and modification of the simulation state.

## The conditions :

Waiting for a cooldown uses ```cond_free``` via ```pthread_cond_wait```, suspending the thread until the release signal (pthread_cond_broadcast).

Cooldown time management uses ```cond_available``` via ```pthread_cond_timedwait```. The thread is put to sleep for the exact duration of the wait time calculated by ```get_end_cooldown```.

Priority conflicts trigger a wait on the ```cond_priority``` condition. The release signal is called when the dongles are released.

## Examples:

![Three coders with 1500 burnout, 100 compiling, 200 cooldown and fifo](https://github.com/Marchombre21/Codexion/raw/main/schemas/codexion-stats(1).png)

Here,the time before burnout is 1500 ms.

**T=0 :** Coder 1 starts by locking key 1 and key 3. Then, since we are in FIFO mode and he is the first to request them, he is granted access and starts compiling.

When coder 2 wanted to take key 1, it was already locked and he had to wait for it to become available. The same applies to coder 3 with key 3.

**T=100 :** At 100 ms, compilation is complete and the keys are released.

**T=300 :** At the end of their cooldown (200 ms), coder 2 was faster than coder 3 and therefore took the keys. Coder 3 returns to wait.

And so on until all have completed the two requested compilations.

![Three coders with 500 burnout, 100 compiling, 200 cooldown and fifo](https://github.com/Marchombre21/Codexion/raw/main/schemas/codexion-stats.png)

Here, the time before burnout is 500 ms.

**T=0 :** Encoder 2 starts by locking key 2 and key 3.
Then, since we are in FIFO mode and it is the first to request them, it is granted access and starts compiling.
When coder 1 wanted to take key 2, it was already locked, and he had to wait for it to be released.
The same applies to coder 3 with key 3.

**T=100 :** At 100 ms, compilation is complete, and the keys are released.

**T=300 :** At the end of their cooldown (200 ms), encoder 3 was faster than encoder 1 and therefore took the keys.
Encoder 1 returns to wait.

**T=500 :** However, the burnout time (500 ms) is reached for encoder 1.
The simulation therefore stops.
